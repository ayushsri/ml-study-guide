Topic 1

What an AI engineer actually does

An AI engineer does not just call models. The real job is turning probabilistic model behavior into a product that users can trust. That means combining data pipelines, model selection, system design, evaluation, product constraints, and production monitoring.

Think in this order: Problem → Data → Baseline → Model choice → System design → Evaluation → Deploy → Monitor.
Remember it as a factory line: first understand the problem, then feed the right data, choose the right machine, check the output, ship it, and keep watching it.
Topic 2

Problem framing

Before building anything, define the task very precisely. Many weak AI systems fail because the team built the wrong thing, not because the model was bad. You should know the exact input, exact output, what counts as success, what errors matter most, and what constraints exist on latency, cost, and reliability.

Ask

  • What is the input?
  • What is the output?
  • What task type is this?

Define

  • Success metric
  • Failure modes
  • Latency/cost budget

Decide

  • Rules vs ML vs LLM
  • Need for retrieval?
  • Need for human review?
Memory line: Input, Output, Success, Failure, Constraints. If those five are clear, the project is much easier.
Topic 3

Classic ML vs LLMs

Classic ML is better when the data is structured, labels exist, tasks are stable, and you need low latency or explainability. LLMs shine when the input is unstructured text and you need flexible language understanding, generation, summarization, extraction, or tool orchestration. RAG becomes useful when answers depend on changing or private knowledge.

Use Classic ML

  • Tabular or numerical data
  • Stable labels
  • Fast, cheap predictions
  • Interpretability matters

Use LLMs

  • Unstructured text
  • Q&A, summarization, extraction
  • Language-heavy workflows
  • Tool-based assistants
Memory trick: Tables → ML, Text → LLM, Private/Fresh knowledge → RAG.
Topic 4

Data is king

Model quality is usually limited by data quality more than model complexity. Poor labels, duplicates, leakage, skew, and stale documents can quietly destroy performance. In RAG systems, retrieval quality is a kind of data quality too, because the model only answers well if the evidence it sees is correct and relevant.

  • Check missing values and inconsistent formats
  • Watch for train/validation/test contamination
  • Check class imbalance and skew
  • In RAG, remove stale or contradictory documents
Memory line: Bad data in, polished bad answers out.
Topic 5

ML fundamentals

These concepts teach engineering judgment. Supervised learning uses labeled data to learn input-output mapping. Unsupervised learning finds hidden structure without labels. Overfitting means the model memorizes noise; underfitting means it is too simple to learn useful patterns. These ideas help you reason about model behavior instead of guessing.

Supervised

Learn from labeled examples like spam detection or churn prediction.

Unsupervised

Find clusters, anomalies, or compressed structure without labels.

Overfitting

Great on training data, weak on new data.

Underfitting

Weak on both training and validation because the model is too simple.

Memory line: supervised learns answers, unsupervised finds patterns, overfitting memorizes, underfitting misses.
Topic 6

Bias-variance tradeoff

Bias is error from being too simple. Variance is error from being too sensitive to the training data. As model complexity goes up, bias often falls and variance often rises. Good generalization comes from balancing both instead of maximizing complexity blindly.

High Bias

Model misses real signal and underfits.

High Variance

Model memorizes noise and overfits.

Memory trick: Bias = too rigid, Variance = too fragile.
Topic 7

Important metrics

Metrics should match the business cost of mistakes. In classification, accuracy can be misleading when classes are imbalanced, so precision, recall, and F1 are often more meaningful. In regression, MAE and RMSE show different error behavior. In retrieval and ranking, metrics like Precision@K, Recall@K, MRR, and NDCG measure whether the right items appear early enough.

Classification

  • Accuracy
  • Precision
  • Recall
  • F1
  • AUC-ROC

Regression

  • MAE
  • MSE / RMSE
  • R²

LLM / Retrieval

  • Faithfulness
  • Groundedness
  • Precision@K
  • Latency
  • Cost/request
Memory line: Choose metrics based on what hurts most when the system is wrong.
Topic 8

Embeddings

Embeddings turn text, images, or other content into dense vectors where similar meaning lands in nearby regions of vector space. They power semantic search, clustering, deduplication, recommendation, and the retrieval stage of RAG. They work well, but only if chunking, metadata, and indexing are handled carefully.

Key idea: closer vectors usually mean closer meaning.
Memory trick: imagine each sentence gets GPS coordinates in meaning-space. Similar ideas live near each other.
Topic 9

Transformers

Transformers process sequences using attention, which lets each token look at other relevant tokens in the sequence. This made them much better than older RNN-style approaches for capturing long-range relationships. They scale well, train in parallel, and form the basis of modern LLMs.

  • Tokenization converts text into tokens
  • Embeddings represent tokens numerically
  • Self-attention learns what each token should focus on
  • Multi-head attention captures different relationships at once
Memory line: a transformer reads by letting every word decide which other words matter most.
Topic 10

Tokenization

Models do not read text as full words the way humans do. They process tokens, which may be full words, pieces of words, punctuation, or symbols. This matters because cost, latency, prompt limits, and chunking strategy all depend on token count, not just visible word count.

Memory trick: the model pays rent per token, not per sentence.
Topic 11

Prompt engineering

Prompting is specifying the behavior you want as clearly as possible. Good prompts define task, context, constraints, output format, and sometimes examples. Great prompt engineering is less about clever tricks and more about reducing ambiguity, constraining the answer shape, and handling common edge cases.

Good Prompt Traits

  • Clear task
  • Explicit constraints
  • Expected output format
  • Relevant examples

Bad Prompt Traits

  • Vague request
  • No format rules
  • Too much irrelevant context
  • No handling of corner cases
Memory line: be clear, be specific, be structured.
Topic 12

Structured outputs

Production systems usually prefer JSON or schema-constrained outputs instead of open-ended text. This makes downstream parsing safer, reduces brittle regex logic, and makes validation possible. But you still need to check for missing keys, wrong types, invalid JSON, or hallucinated fields.

Memory trick: free text is easy to read, structured output is easy to build with.
Topic 13

Tool calling

Tool calling lets the model use external capabilities such as search, calculators, APIs, databases, or code execution. The typical pattern is: user asks, model decides the tool, the tool runs, then the model synthesizes the final answer. The main risks are wrong tool choice, bad arguments, silent failures, and looping behavior.

Memory line: the LLM decides, the tool does, the LLM explains.
Topic 14

RAG

RAG means Retrieve, then Generate. Instead of relying only on built-in model memory, the system first fetches relevant external knowledge and then asks the model to answer using that evidence. This helps with fresher knowledge, private knowledge, citations, and lower hallucination risk.

Pipeline

  • Ingest docs
  • Chunk docs
  • Embed chunks
  • Store in vector DB
  • Retrieve
  • Rerank
  • Generate

Why use it

  • Current information
  • Private documents
  • Evidence-backed answers
  • Reduced hallucination

Failure points

  • Bad chunking
  • Noisy docs
  • Poor retrieval
  • Weak grounding prompt
Memory line: RAG brings the book before the model writes the answer.
Topic 15

Fine-tuning vs RAG

Use RAG when you need knowledge injection from private or changing documents. Use fine-tuning when you need the model to behave differently: style consistency, task specialization, domain patterns, or better response format behavior. Fine-tuning is usually not the right tool for constantly changing facts.

Memory trick: RAG teaches facts at runtime; fine-tuning teaches habits.
Topic 16

Hallucination

Hallucination happens when the model produces information that is unsupported or false. It often comes from missing context, ambiguous prompts, weak retrieval, or asking the model to go beyond available evidence. Good systems reduce hallucination by grounding, citing sources, validating outputs, and abstaining when evidence is missing.

Memory line: when evidence is weak, the model may guess.
Topic 17

AI agents

An agent is an LLM-based system that can plan, take actions, use tools, and adapt based on intermediate results. Agents are useful for multi-step workflows but are often overused. If a simple prompt or deterministic workflow works, an agent may add cost, latency, and debugging pain without enough benefit.

Good for

  • Multi-step planning
  • Tool use
  • Environment feedback
  • Replanning

Risks

  • Loops
  • Cost explosion
  • Hidden failures
  • Harder debugging
Memory trick: use agents only when the task really has multiple moving parts.
Topic 18

Evaluation

If you cannot measure a system, you cannot trust or improve it. Offline evaluation checks behavior before deployment. Online evaluation measures real user outcomes in production. Human evaluation is essential when correctness depends on quality, style, usefulness, or faithfulness rather than exact labels alone.

Offline

Benchmarks and test suites before launch.

Online

Live metrics like task success, latency, and abandonment.

Human

Rubric-based judgment for nuanced quality.

Memory line: offline tells you if it might work, online tells you if it does work.
Topic 19

LLM-as-judge

This means using one model to score or compare outputs from another model. It is helpful for large-scale evaluation and rubric-based reviews, but it can also introduce bias or inconsistency. The safest setup is LLM judge plus human audits plus task-grounded metrics.

Memory trick: a model judge is a fast reviewer, not the final court.
Topic 20

Guardrails

Guardrails are controls that improve both safety and reliability. Examples include input validation, schema validation, rate limits, output filtering, tool restrictions, retrieval filters, and fallback behavior. Guardrails keep probabilistic systems from behaving unpredictably in production.

Memory line: guardrails keep the model on the road.
Topic 21

Latency optimization

Latency often comes from large prompts, slow retrieval, multiple model calls, tool chains, and overly long outputs. You can reduce it by shrinking token counts, choosing smaller models where possible, parallelizing independent steps, caching repeated work, and avoiding unnecessary agent loops.

Memory trick: every extra token and every extra call adds waiting time.
Topic 22

Cost optimization

The biggest cost drivers are prompt tokens, completion tokens, the number of calls, and the model tier you choose. Cost comes down when you cache, trim unnecessary history, retrieve fewer but better chunks, route easy tasks to cheaper models, and keep outputs appropriately short.

Memory line: cost = tokens × calls × model price.
Topic 23

Caching

Caching stores expensive intermediate or final results so repeated work can be skipped. Common cache layers include prompt-response cache, retrieval cache, embedding cache, and feature cache. Caching saves time and money, but only if you also manage invalidation when the knowledge or prompt logic changes.

Memory trick: cache saves repeated thinking, but stale cache can lie.
Topic 24

Observability

Observability means having enough logging and tracing to understand why the system behaved a certain way. For AI systems, that includes prompts, retrieved chunks, tool calls, model outputs, token usage, latency by stage, and user feedback. Without observability, debugging becomes guesswork.

Memory line: if you cannot see inside the pipeline, you cannot fix it.
Topic 25

Common production failure modes

AI systems often fail quietly. Common issues include wrong problem framing, weak eval sets, stale retrieval corpora, malformed structured output, silent tool failures, high latency, high cost, prompt regressions, and missing fallbacks or monitoring.

  • Wrong task definition
  • Retrieval returning stale or irrelevant context
  • Parsing failures from malformed JSON
  • No rollback or no alerting when quality drops
Memory trick: most failures are system failures, not pure model failures.
Topic 26

Good system design thinking

A strong AI engineer keeps asking practical questions: what is the simplest baseline, where does truth come from, how do we evaluate, what happens when the model fails, and how do we rollback changes safely. This mindset matters more than using trendy frameworks.

Memory line: design for failure first, then for success.
Topic 27

Model selection

The best model is not the biggest one. You choose based on task difficulty, latency budget, cost budget, accuracy requirement, context length needs, output structure needs, and safety requirements. Strong engineers match model capacity to business need.

Memory trick: right-sized beats oversized.
Topic 28

Retrieval design notes

Retrieval quality depends on chunk size, overlap, metadata, hybrid search, reranking, query rewriting, top-k choice, and document cleaning. Dense retrieval finds semantic similarity, keyword retrieval preserves exact matches, and reranking helps refine the best candidates.

A strong practical pattern is: dense retrieval + keyword retrieval + reranking.
Memory line: retrieve broadly, then refine sharply.
Topic 29

Data leakage

Leakage happens when the model sees information during training or evaluation that would not be available at real inference time. This creates fake confidence. Common examples include future information leaking into features, duplicates across train and test, or labels indirectly appearing in inputs.

Memory trick: if the model knows tomorrow while predicting today, you have leakage.
Topic 30

Feature engineering still matters

Even in the deep learning era, feature design and preprocessing still matter a lot, especially in tabular problems and retrieval systems. Good features help the model see the useful signal more clearly. In many practical cases, feature engineering beats chasing a fancier algorithm.

Memory line: better inputs often beat smarter models.
Topic 31

Deployment mindset

Shipping a model is the start, not the finish. After deployment, you need monitoring, drift checks, refreshed evaluations, prompt/version control, rollback paths, incident handling, and user feedback loops. A production AI system is a living system.

Memory trick: deployed does not mean done.
Topic 32

Drift

Drift means the world changed after deployment. Data drift means the input distribution changed. Concept drift means the relationship between input and target changed. Retrieval drift means the document corpus changed. User behavior drift means people now use the system differently than before. All of these require periodic re-evaluation.

Data drift

Inputs look different now.

Concept drift

The rules connecting input to truth changed.

Retrieval / behavior drift

Docs or usage patterns changed.

Memory line: something changed after launch. Ask: data, truth, docs, or usage?
Topic 33

Human-in-the-loop

Human review is useful when decisions are high-stakes, confidence is low, or output quality must be approved before action. Good HITL systems do more than add humans randomly; they route the right cases to humans and log feedback so the system can improve over time.

Memory trick: let the model draft, let the human decide when risk is high.
Topic 34

Security / safety notes

AI systems face prompt injection, data exfiltration, unsafe tool usage, secret leakage, jailbreak attempts, and over-permissioned agent behavior. Good defenses include least privilege, tool isolation, strict argument validation, content sanitization, and never blindly trusting instructions found inside retrieved documents.

Memory line: treat model inputs as untrusted, especially retrieved text.
Topic 35

Good engineering habits

The habits that create strong AI engineers are simple but powerful: start with a baseline, measure before optimizing, compare alternatives empirically, build modular pipelines, keep prompts versioned, log important events, and document assumptions clearly.

Memory trick: simple baseline first, evidence before opinions.
Topic 36

Interview-style one-liners

Good AI engineer

Builds reliable AI systems, not just model demos.

What matters in RAG

Retrieval quality, chunking, reranking, grounding, evaluation.

Why systems fail

Poor data, weak eval, bad retrieval, missing guardrails.

Memory line: reliability beats flashiness.
